日誌系統完全版
TRACE · DEBUG · INFO · WARN · ERROR · Logback配置 · 使用示例
① 日誌級別金字塔 — 從低到高
← 信息量最多，性能消耗最大
TRACE 最詳細的追蹤信息，方法進出、每個變量值 Level 1
↑ 包含下面所有級別
DEBUG 調試信息，開發時用 Level 2
↑
INFO 正常運行信息，生產環境標準 Level 3
↑
WARN 警告，可能有問題但不影響運行 Level 4
↑
ERROR 錯誤，需要立即關注 Level 5
← 信息量最少，最需要關注
💡 級別的繼承關係：設置為INFO意味著INFO、WARN、ERROR都會輸出，TRACE和DEBUG被過濾掉。
設置為WARN意味著只輸出WARN和ERROR。級別越高，輸出越少，性能越好。
② 五個級別詳解 — 什麼時候用哪個？
TRACE 極少使用
使用場景

追蹤方法的每一步執行，記錄所有輸入輸出參數。框架層面的底層調試。
一般不在業務代碼中使用。

示例
log.trace("進入findById方法, id={}", id); log.trace("SQL執行完成, 返回結果: {}", result); log.trace("退出findById方法");
DEBUG 開發環境
使用場景

開發和測試時的調試信息。記錄關鍵變量值、方法參數、中間計算結果。
生產環境關閉，避免日誌量過大。

示例
log.debug("查詢用戶, id={}", userId); log.debug("分頁參數: page={}, size={}", pageNum, pageSize); log.debug("計算結果: totalAmount={}", amount);
INFO 生產環境✅
使用場景

記錄系統正常運行的重要事件。
服務啟動/停止、用戶登錄、訂單創建、定時任務執行等業務里程碑。

示例
log.info("用戶登錄成功, userId={}", userId); log.info("訂單創建成功, orderId={}", orderId); log.info("定時任務開始執行: {}", taskName); log.info("服務啟動完成, port={}", port);
WARN 需要關注⚠️
使用場景

潛在問題，當前可以繼續運行但需要關注。
重試操作、接近閾值、使用了廢棄API、業務異常（用戶不存在等）。

示例
log.warn("業務異常: code={}, msg={}", code, msg); log.warn("接口重試第{}次, url={}", retryCount, url); log.warn("緩存未命中, key={}", cacheKey); log.warn("內存使用率已達{}%", memoryUsage);
ERROR 立即處理🚨
使用場景

系統錯誤，影響功能正常使用。
未捕獲異常、數據庫連接失敗、第三方接口異常、系統崩潰。必須記錄完整堆疊！

示例
// 第二個參數傳入Exception，記錄完整堆疊 log.error("系統異常", e); log.error("數據庫連接失敗, url={}", dbUrl, e); log.error("支付接口調用失敗, orderId={}", orderId, e);
③ 不同環境的日誌級別配置

💻 開發環境（dev）

DEBUG

看到最多信息，方便調試。SQL語句、參數、返回值全部輸出。

輸出：TRACE DEBUG INFO WARN ERROR

🧪 測試環境（test）

INFO

模擬生產環境的日誌量，驗證功能正確性，不需要調試細節。

輸出：INFO WARN ERROR

🏭 生產環境（prod）

WARN

只記錄警告和錯誤，減少磁盤IO，保持高性能。敏感信息不輸出。

輸出：WARN ERROR
④ 代碼中如何使用日誌
📦 引入方式（推薦Lombok）
// 方式1：使用Lombok @Slf4j（推薦，最簡潔） @Slf4j ← 自動創建 log 變量 @Service public class UserService { // 直接使用 log.xxx() 即可，不需要手動聲明 } // 方式2：手動聲明（不用Lombok） public class UserService { private static final Logger log = LoggerFactory.getLogger(UserService.class); }
⭐ 完整Service層使用示例
@Slf4j @Service public class OrderService { public Order createOrder(OrderDTO dto) { log.debug("開始創建訂單, userId={}, amount={}", dto.getUserId(), dto.getAmount()); // 1. 驗證用戶 User user = userService.findById(dto.getUserId()); log.debug("用戶驗證通過, userName={}", user.getName()); // 2. 檢查庫存 if (stock < dto.getQuantity()) { log.warn("庫存不足, productId={}, stock={}, required={}", productId, stock, dto.getQuantity()); throw new BusinessException(ErrorCode.INSUFFICIENT_STOCK); } // 3. 創建訂單 Order order = orderMapper.insert(newOrder); log.info("訂單創建成功, orderId={}, userId={}, amount={}", order.getId(), dto.getUserId(), dto.getAmount()); // 4. 調用支付（可能失敗） try { paymentService.pay(order); } catch (Exception e) { log.error("支付服務調用失敗, orderId={}", order.getId(), e); throw new BusinessException(ErrorCode.PAYMENT_FAILED); } return order; } }
⚠️ ERROR日誌的正確寫法
// ❌ 錯誤：只記錄message，丟失堆疊信息 log.error("支付失敗: " + e.getMessage()); // ✅ 正確：把Exception作為最後一個參數，記錄完整堆疊 log.error("支付服務調用失敗, orderId={}", orderId, e); // ✅ 正確：記錄關鍵上下文 + 完整異常 log.error("數據庫操作失敗, table={}, operation={}", tableName, operation, e); // ❌ 錯誤：用String拼接（性能差，應用占位符{}） log.debug("用戶ID是: " + userId + ", 名稱是: " + userName); // ✅ 正確：用{}占位符（未輸出時不做字串拼接，性能好） log.debug("用戶: id={}, name={}", userId, userName);
⑤ 日誌配置（application.yml）
⚙️ application.yml
logging: level: root: WARN # 全局默認級別 com.example: INFO # 自己的包用INFO com.example.service: DEBUG # Service層用DEBUG（開發時） org.springframework: WARN # Spring框架本身WARN就夠 org.mybatis: DEBUG # MyBatis輸出SQL（開發時） file: name: logs/application.log # 日誌輸出到文件 pattern: console: "%d{HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n" file: "%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger - %msg%n" # 多環境配置（通過spring.profiles.active切換） --- spring.config.activate.on-profile: dev logging.level: com.example: DEBUG org.mybatis: DEBUG # 開發環境看SQL --- spring.config.activate.on-profile: prod logging.level: root: WARN com.example: INFO # 生產環境只看INFO以上
⑥ 日誌輸出格式解析
// 一條典型的日誌輸出： 2024-03-15 10:30:45.123 [http-nio-8080-exec-1] INFO c.e.service.UserService - 訂單創建成功, orderId=101, userId=1, amount=350.00 // 各部分說明： 2024-03-15 10:30:45.123 ← 時間戳（精確到毫秒） [http-nio-8080-exec-1] ← 線程名（哪個線程處理的請求） INFO ← 日誌級別 c.e.service.UserService ← 類名縮寫（com.example.service.UserService） 訂單創建成功, orderId=101... ← 日誌消息 // ERROR帶堆疊的輸出： 2024-03-15 10:30:50.456 [exec-2] ERROR c.e.service.OrderService - 支付服務調用失敗, orderId=102 java.net.ConnectException: Connection refused at com.example.service.PaymentService.pay(PaymentService.java:45) at com.example.service.OrderService.createOrder(OrderService.java:78) ...
💡 三個最佳實踐：
① 用{}占位符不用字串拼接 → 未輸出時不消耗性能
② ERROR必須帶Exception參數 → 保留完整堆疊，排查問題的唯一線索
③ 生產環境設WARN或INFO → 減少磁盤IO，INFO級別的日誌量已經夠用
