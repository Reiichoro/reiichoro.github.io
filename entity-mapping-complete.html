💡 ORM（物件關聯映射）的核心思想：
一個Java類 ↔ 一張數據庫表 ／ 一個Java物件 ↔ 表中的一條記錄 ／ 類的屬性 ↔ 表的欄位
開發者操作Java物件，框架自動生成對應SQL，不需要手寫SQL。
// 完整實體類範例（使用Lombok + JPA注解） @Entity ← 標識這是數據庫實體 @Table(name = "users") ← 對應數據庫表名 @Data ← Lombok自動生成Getter/Setter @NoArgsConstructor ← 自動生成無參構造函數 public class User { @Id ← 主鍵標識 @GeneratedValue(strategy = GenerationType.IDENTITY) ← 自動遞增 private Long id; @Column(name = "name", nullable = false, length = 50) private String name; @Column(unique = true) ← 唯一約束 private String email; @Column(name = "created_at") ← 欄位名和屬性名不同時使用 private LocalDateTime createdAt; }
② 關係映射 — 表與表之間的關聯

1️⃣ 一對一（One-to-One）

User ──── UserProfile
一個用戶 對應 一份資料
@OneToOne
例：用戶 ↔ 用戶詳細資料
通常用外鍵或共享主鍵實現
@OneToOne(cascade = CascadeType.ALL) @JoinColumn(name = "profile_id") private UserProfile profile;

2️⃣ 一對多（One-to-Many）

User ──── List<Order>
一個用戶 對應 多個訂單
@OneToMany / @ManyToOne
例：用戶 ↔ 訂單（最常見）
外鍵放在「多」的那一側
// User端 @OneToMany(mappedBy = "user") private List<Order> orders; // Order端 @ManyToOne @JoinColumn(name = "user_id") private User user;

3️⃣ 多對多（Many-to-Many）

User ──── Role
多個用戶 可以有 多個角色
@ManyToMany
例：用戶 ↔ 角色（權限系統）
需要中間關聯表 user_role
@ManyToMany @JoinTable(name = "user_role", joinColumns = @JoinColumn(name="user_id"), inverseJoinColumns = @JoinColumn(name="role_id")) private List<Role> roles;

4️⃣ 懶加載 vs 急加載

LAZY（懶）vs EAGER（急）
何時從數據庫加載關聯數據？
FetchType.LAZY（默認・推薦）
→ 用到時才查詢，節省性能

FetchType.EAGER
→ 主實體查詢時立刻一起查
→ 可能導致N+1問題，謹慎使用
⚠️ N+1問題：查詢10個用戶，每個用戶又各自查一次訂單 = 執行了11次SQL。
解決方案：使用 JOIN FETCH 或 MyBatis的一次性聯表查詢。

④ 數據訪問層模式 — Repository vs DAO vs 其他

📁 Repository模式（Spring推薦）

將數據訪問邏輯封裝在Repository接口中，繼承JpaRepository或自定義。
強調「集合」的概念，像操作內存中的集合一樣操作數據庫。

Spring Data JPA
public interface UserRepository extends JpaRepository<User, Long> { // 自動實現：findById、save、delete等 List<User> findByName(String name); }

🗃️ DAO模式（Data Access Object）

傳統模式，定義接口描述所有數據操作，再提供實現類。
MyBatis的Mapper本質上就是DAO模式。職責單一，測試容易。

MyBatis Mapper
@Mapper public interface UserMapper { User findById(Long id); void insert(User user); void update(User user); void deleteById(Long id); }

🔄 Active Record模式

實體類本身包含數據庫操作方法，物件可以自己保存、刪除。
Ruby on Rails的風格，Java中較少見（jOOQ支持）。

較少使用
// 物件自己操作數據庫（非主流） User user = new User(); user.name = "Alice"; user.save(); ← 物件自己存入數據庫 user.delete();

🔧 Query Builder模式

用Java代碼動態構建SQL查詢，避免字串拼接的安全問題。
MyBatis-Plus的LambdaQueryWrapper就是這種模式。

MyBatis-Plus
// 動態構建查詢條件（MyBatis-Plus） LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>() .eq(User::getName, "Alice") .gt(User::getAge, 18) .orderByDesc(User::getCreatedAt);
