SecurityConfig 深度解析
@Configuration · 路徑規則 · 密碼加密 · 異常處理 · 完整配置
① 類級別注解解析
@Configuration

告訴Spring這是一個配置類，裡面的@Bean方法會被Spring管理。
相當於XML配置的替代方案。

@EnableMethodSecurity

開啟方法級別的權限控制。
沒有這個注解，@PreAuthorize就不生效！
必加。

@Bean

方法返回的物件交由Spring容器管理。
SecurityFilterChain、PasswordEncoder等都通過@Bean注冊。

@Slf4j

Lombok提供的日誌注解，自動創建log變量。
可以記錄配置加載、認證失敗等關鍵事件。

@Configuration ← 這是Spring配置類 @EnableMethodSecurity ← 開啟@PreAuthorize等方法級別權限 @Slf4j ← 日誌（可選） public class SecurityConfig { // 所有Security相關的Bean都在這裡配置 }

return http.build(); } // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ // Bean②：密碼加密器（BCrypt是業界標準） // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ @Bean public PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); // BCrypt：每次加密結果不同，防彩虹表攻擊 // 用法：encoder.encode("明文") 和 encoder.matches("明文", "密文") } // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ // Bean③：認證管理器（登錄時使用） // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ @Bean public AuthenticationManager authenticationManager( AuthenticationConfiguration config) throws Exception { return config.getAuthenticationManager(); } // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ // Bean④：跨域配置 // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ @Bean public CorsConfigurationSource corsConfigurationSource() { CorsConfiguration config = new CorsConfiguration(); UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource(); source.registerCorsConfiguration("/**", config); return source; } }
③ 路徑規則方法速查
.permitAll()
所有人都可以訪問，不需要登錄
/api/auth/login /api/public/**
.authenticated()
必須已登錄（有有效Token），不限角色
anyRequest().authenticated()
.hasRole("ADMIN")
必須有指定角色。注意：Spring自動加ROLE_前綴
hasRole("ADMIN") → 實際匹配 ROLE_ADMIN
/api/admin/**
.hasAnyRole("A","B")
有其中任意一個角色即可訪問
hasAnyRole("ADMIN", "MANAGER")
.hasAuthority("perm")
匹配具體的權限字串，不會自動加ROLE_前綴
更細粒度的權限控制，如 user:delete、order:view
hasAuthority("user:delete")
.denyAll()
所有人都拒絕，通常用於臨時關閉某接口
requestMatchers("/api/deprecated/**").denyAll()
⚠️ 路徑規則的順序很重要！
Spring Security從上到下依次匹配，第一個匹配的規則生效。
越具體的路徑放前面，anyRequest()必須放最後。
如果把 anyRequest().permitAll() 放在最前面，後面所有規則都失效！
④ 路徑匹配規則（requestMatchers的寫法）
寫法匹配範圍示例
/api/login 精確匹配，只匹配這一個路徑 只匹配 /api/login
/api/users/** ** 匹配任意深度路徑 /api/users/1, /api/users/1/orders 都匹配
/api/users/* * 只匹配單層路徑 /api/users/1 匹配，/api/users/1/orders 不匹配
HttpMethod.GET, "/api/**" 指定HTTP方法+路徑 只有GET請求的/api/**才匹配
anyRequest() 匹配所有請求（兜底） 必須放最後，覆蓋前面未匹配的
⑤ SecurityConfig路徑規則 vs @PreAuthorize — 兩種控制方式

🛡️ SecurityConfig路徑規則

// 在SecurityConfig中統一配置 .requestMatchers("/api/admin/**") .hasRole("ADMIN")

✅ 集中管理，一目了然
✅ 在過濾器層就攔截，效率高
✅ 適合粗粒度的路徑級別控制
❌ 不能做複雜的業務邏輯判斷

🔐 @PreAuthorize方法注解

// 直接在方法上加注解 @PreAuthorize("hasRole('ADMIN') and #id == authentication.principal") public void deleteUser(Long id) { }

✅ 可以訪問方法參數（#id）
✅ 支持SpEL表達式，靈活強大
✅ 適合細粒度的業務邏輯控制
✅ 需要@EnableMethodSecurity開啟

💡 最佳實踐：兩種方式搭配使用
SecurityConfig → 控制粗粒度路徑（/admin/**需要ADMIN）
@PreAuthorize → 控制細粒度業務（只能刪除自己的資源）
⑥ UserDetailsService — 從數據庫加載用戶
👤 UserDetailsService實現
@Service public class UserDetailsServiceImpl implements UserDetailsService { @Autowired private UserMapper userMapper; @Override public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException { // 從數據庫查詢用戶 User user = userMapper.findByUsername(username); if (user == null) { throw new UsernameNotFoundException("用戶不存在: " + username); } // 轉換角色為GrantedAuthority List<GrantedAuthority> authorities = user.getRoles().stream() .map(role -> new SimpleGrantedAuthority("ROLE_" + role)) .collect(Collectors.toList()); // 返回Spring Security的UserDetails物件 return new org.springframework.security.core.userdetails.User( user.getUsername(), user.getPassword(), ← 必須是BCrypt加密後的密碼 authorities ); } }
🔑 登錄時如何使用AuthenticationManager
@RestController @RequestMapping("/api/auth") public class AuthController { @Autowired private AuthenticationManager authenticationManager; @Autowired private JwtUtils jwtUtils; @PostMapping("/login") public Result<String> login(@RequestBody LoginDTO dto) { try { // 1. 調用AuthenticationManager驗證用戶名密碼 // 內部會調用UserDetailsService.loadUserByUsername // 再用BCrypt比對密碼 Authentication authentication = authenticationManager.authenticate( new UsernamePasswordAuthenticationToken( dto.getUsername(), dto.getPassword() ) ); // 2. 認證成功，取出用戶信息 UserDetails userDetails = (UserDetails) authentication.getPrincipal(); // 3. 生成JWT Token String token = jwtUtils.generateToken(userDetails); return Result.success(token); } catch (BadCredentialsException e) { throw new BusinessException(ErrorCode.LOGIN_FAILED, "用戶名或密碼錯誤"); } } }
⑦ 密碼加密 — BCrypt是業界標準

❌ 絕對不能這樣做

// 明文儲存密碼（嚴重安全漏洞！） user.setPassword("123456"); // MD5加密（已被破解，不安全） user.setPassword(MD5.encode("123456"));

✅ 正確做法：BCrypt

// 注冊時加密保存 String encoded = passwordEncoder.encode("123456"); user.setPassword(encoded); // 登錄時驗證（Spring Security自動做） boolean ok = passwordEncoder.matches("123456", encodedPassword);
// BCrypt的特點： // 同一個密碼每次加密結果都不同！ passwordEncoder.encode("123456") // 第一次：$2a$10$abcdefghijklmnopqrstuuV9mJ... // 第二次：$2a$10$XYZabcdefghijklmnopqrstuu... // 兩次結果完全不同！但matches()都能驗證成功 // 原因：鹽值（salt）被內嵌在密文中，每次隨機生成
