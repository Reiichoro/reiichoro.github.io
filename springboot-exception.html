✅ 統一處理（整潔）

// 前端永遠收到固定格式 { "code": 404, "message": "用戶不存在", "data": null, "timestamp": "2024-03-15T10:30:00" } // 成功也是固定格式 { "code": 200, "message": "success", "data": { ... } }
💡 統一異常處理的三大好處：
① 前端只需要判斷一種格式，不用處理各種奇怪的錯誤結構
② 敏感的堆疊信息不會暴露給用戶（安全性）
③ 日誌集中記錄，方便排查問題
② 自定義異常體系 — 異常階層設計
RuntimeException
Java內建，所有運行時異常的父類。不需要強制try-catch。
BaseException（自定義基礎類）
繼承RuntimeException，添加 code（錯誤碼）和 message（錯誤信息）兩個屬性。所有業務異常繼承它。
BusinessException（業務異常）
業務邏輯錯誤：用戶不存在、餘額不足、訂單已取消等。HTTP狀態碼通常是4xx。
ValidationException（參數驗證異常）
請求參數不合法：名稱為空、金額為負數、日期格式錯誤等。HTTP狀態碼400。
💥 自定義異常類代碼
// 1. 錯誤碼枚舉（統一管理所有錯誤） public enum ErrorCode { SUCCESS(200, "成功"), BAD_REQUEST(400, "請求參數錯誤"), UNAUTHORIZED(401, "未登錄或Token過期"), FORBIDDEN(403, "無權限訪問"), NOT_FOUND(404, "資源不存在"), SERVER_ERROR(500, "服務器內部錯誤"), // 業務錯誤碼（自定義，通常4位數） USER_NOT_FOUND(1001, "用戶不存在"), ORDER_NOT_FOUND(1002, "訂單不存在"), INSUFFICIENT_BALANCE(1003, "餘額不足"), ORDER_ALREADY_CANCELLED(1004, "訂單已取消"); private final int code; private final String message; // constructor + getter... } // 2. 基礎異常類 public class BaseException extends RuntimeException { private final int code; public BaseException(ErrorCode errorCode) { super(errorCode.getMessage()); this.code = errorCode.getCode(); } public BaseException(ErrorCode errorCode, String message) { super(message); ← 自定義消息覆蓋默認 this.code = errorCode.getCode(); } } // 3. 業務異常（直接使用這個） public class BusinessException extends BaseException { public BusinessException(ErrorCode errorCode) { super(errorCode); } public BusinessException(ErrorCode errorCode, String message) { super(errorCode, message); } }
③ 異常拋出點 — 在哪裡throw？
⚙️ Service層（最主要的拋出點）
@Service public class UserService { public User findById(Long id) { User user = userMapper.findById(id); // ⭐ 異常拋出點1：業務條件不滿足 if (user == null) { throw new BusinessException(ErrorCode.USER_NOT_FOUND); } return user; } public void cancelOrder(Long orderId) { Order order = orderMapper.findById(orderId); // ⭐ 異常拋出點2：狀態不符合業務規則 if ("cancelled".equals(order.getStatus())) { throw new BusinessException( ErrorCode.ORDER_ALREADY_CANCELLED, "訂單" + orderId + "已取消，無法再次取消" ); } } public void pay(Long userId, BigDecimal amount) { BigDecimal balance = getBalance(userId); // ⭐ 異常拋出點3：數據不滿足條件 if (balance.compareTo(amount) < 0) { throw new BusinessException(ErrorCode.INSUFFICIENT_BALANCE); } } }
🌐 Controller層（參數驗證異常）
@RestController public class UserController { // 使用@Valid自動觸發參數驗證異常 @PostMapping("/users") public Result createUser(@RequestBody @Valid UserDTO dto) { // @Valid會自動驗證，不需要手動throw // 驗證失敗 → 自動拋出 MethodArgumentNotValidException return Result.success(userService.create(dto)); } } // UserDTO的驗證注解 @Data public class UserDTO { @NotBlank(message = "姓名不能為空") @Length(max = 50, message = "姓名不超過50字") private String name; @Email(message = "Email格式不正確") private String email; @Min(value = 0, message = "年齡不能為負數") private Integer age; }
④ 異常傳播路徑 — 從拋出到捕獲

異常從拋出到全局處理器的完整旅程

Service層
throw new BusinessException(ErrorCode.USER_NOT_FOUND)
異常在Service的findById方法中拋出
↓ 異常向上傳播（不被攔截）
Controller層
Controller調用了Service，但沒有try-catch，異常繼續向上拋
↓ 異常繼續向上傳播
@ControllerAdvice（全局處理器）
攔截！ Spring框架把異常路由到對應的 @ExceptionHandler 方法
↓ 處理後返回統一格式
前端收到統一JSON響應
{"code":1001,"message":"用戶不存在","data":null}
⚠️ 關鍵原則：Service和Controller層不要try-catch業務異常，讓它自然傳播到全局處理器。只有在需要「異常後繼續執行」的場景才在局部catch。
⑤ 全局異常處理器 — 最核心的代碼
🌍 GlobalExceptionHandler
@ControllerAdvice ← 全局異常處理器標識 @Slf4j ← Lombok日誌注解 public class GlobalExceptionHandler { // ① 處理業務異常（BusinessException） @ExceptionHandler(BusinessException.class) @ResponseBody public Result<?> handleBusinessException(BusinessException e) { log.warn("業務異常: code={}, message={}", e.getCode(), e.getMessage()); return Result.error(e.getCode(), e.getMessage()); } // ② 處理參數驗證異常（@Valid觸發） @ExceptionHandler(MethodArgumentNotValidException.class) @ResponseBody public Result<?> handleValidException(MethodArgumentNotValidException e) { // 提取第一個驗證錯誤信息 String message = e.getBindingResult() .getFieldErrors() .stream() .map(FieldError::getDefaultMessage) .findFirst() .orElse("參數驗證失敗"); log.warn("參數驗證失敗: {}", message); return Result.error(400, message); } // ③ 處理未授權異常 @ExceptionHandler(UnauthorizedException.class) @ResponseBody public Result<?> handleUnauthorized(UnauthorizedException e) { return Result.error(401, "未登錄或Token過期，請重新登錄"); } // ④ 兜底：處理所有未預料的異常（最後一道防線） @ExceptionHandler(Exception.class) @ResponseBody public Result<?> handleException(Exception e) { // 未知異常要記錄完整堆疊，方便排查 log.error("系統異常", e); // 不把詳細錯誤暴露給前端（安全性！） return Result.error(500, "系統繁忙，請稍後重試"); } }
⑥ 統一返回格式 — Result類
✅ Result工具類
@Data @AllArgsConstructor public class Result<T> { private Integer code; ← 狀態碼 private String message; ← 提示信息 private T data; ← 響應數據（泛型） private LocalDateTime timestamp; ← 響應時間 // ── 靜態工廠方法（便捷使用）── // 成功（帶數據） public static <T> Result<T> success(T data) { return new Result<>(200, "success", data, LocalDateTime.now()); } // 成功（不帶數據，如刪除操作） public static Result<Void> success() { return new Result<>(200, "success", null, LocalDateTime.now()); } // 失敗（自定義code和message） public static <T> Result<T> error(int code, String message) { return new Result<>(code, message, null, LocalDateTime.now()); } // 失敗（使用ErrorCode枚舉） public static <T> Result<T> error(ErrorCode errorCode) { return new Result<>( errorCode.getCode(), errorCode.getMessage(), null, LocalDateTime.now() ); } }

💡 業務錯誤碼 vs HTTP狀態碼：
HTTP狀態碼是協議層面的（4xx客戶端錯誤，5xx服務器錯誤）
業務錯誤碼是應用層面的（1001用戶不存在，1002訂單不存在）
實際開發中，HTTP狀態碼統一返回200，靠業務code來區分具體錯誤——這樣前端處理更統一。
